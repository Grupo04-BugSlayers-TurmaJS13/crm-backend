# Como Contribuir para o Conecta CRM

Olá! Ficamos felizes por você ter interesse em contribuir com o projeto Conecta CRM! Toda contribuição é bem-vinda.

Este documento estabelece algumas diretrizes para contribuições, garantindo que o processo seja o mais tranquilo possível para todos.

## Reportando Bugs

Se você encontrar um bug, por favor, abra uma **Issue** no nosso repositório do GitHub. Ao reportar, inclua o máximo de detalhes possível:

-   Uma descrição clara e concisa do que é o bug.
-   Passos para reproduzir o comportamento.
-   O comportamento esperado.
-   Screenshots, se aplicável, para ajudar a ilustrar o problema.

## Sugerindo Melhorias

Se você tem uma ideia para uma nova funcionalidade ou uma melhoria para uma já existente, sinta-se à vontade para abrir uma **Issue** para que possamos discutir.

-   Descreva a sua ideia em detalhes.
-   Explique por que essa melhoria seria útil para o projeto.

## Processo de Contribuição

1.  **Faça um Fork** do repositório para a sua conta do GitHub.
2.  **Clone o seu fork** para a sua máquina local (`git clone https://github.com/seu-usuario/crm-backend.git`).
3.  **Crie uma Branch** para a sua modificação (`git checkout -b feature/sua-feature`). Use um nome descritivo para a branch.
4.  **Faça as suas alterações.** Certifique-se de que o código segue o estilo e as convenções do projeto.
5.  **Adicione testes** para as suas alterações, se aplicável.
6.  **Faça o Commit** das suas alterações com uma mensagem clara e seguindo as [Convenções de Commit](#convenções-de-commit) (`git commit -m 'feat: Descrição da sua feature'`).
7.  **Envie para a sua Branch** no seu fork (`git push origin feature/sua-feature`).
8.  **Abra um Pull Request** no repositório original do `crm-backend`.

## Convenções de Commit

Nós utilizamos a convenção **Conventional Commits**. Isso nos ajuda a manter um histórico de commits legível e a automatizar o versionamento.

O formato é: `<tipo>(<escopo>): <descrição>`

-   **Tipos comuns:**
    -   `feat`: Uma nova funcionalidade.
    -   `fix`: Uma correção de bug.
    -   `docs`: Alterações na documentação.
    -   `style`: Alterações que não afetam o significado do código (espaçamento, formatação, etc).
    -   `refactor`: Uma alteração de código que não corrige um bug nem adiciona uma funcionalidade.
    -   `test`: Adicionando testes ou corrigindo testes existentes.
    -   `chore`: Alterações em builds, dependências, etc.
